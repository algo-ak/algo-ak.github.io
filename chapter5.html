Chapter 05

Recursion and Divide & Conquer

Chapters 3 and 4 analysed loops. This chapter analyses functions that call themselves: what recursion is, how its cost becomes a recurrence relation, and three ways to solve recurrences — substitution, recursion trees, and the Master theorem (in its classic, decreasing, and dividing forms). We close with the design paradigm built on recursion: divide and conquer.

5.1Functions — a Quick Refresher

Definition · Function A reusable block of code that performs a specific task: takes input, processes it, and may return an output. 
def function(arguments):
    steps
    return result

Why functions are used:

  • Code reusability — write once, use many times.
  • Modularity — break a large problem into smaller parts.
  • Readability — the program becomes easier to understand.
  • Maintainability — easy to debug and modify.

5.2Recursive Functions

Definition · Recursive function A function that calls itself. 
def recursive_function(parameters):
    if base_case_condition:
        return result
    else:
        return recursive_function(smaller_problem)

Key features

  • Base case(s) — to terminate the recursion.
  • Recursive case(s) — to break the problem into smaller instances.
No base case? Without a reachable base case the function calls itself forever — in practice the call stack overflows and the program crashes. Every recursive design starts by asking: what is the smallest input I can answer directly?

Applications

Factorial computation, the Fibonacci sequence, tree traversals, and many more.

5.3Example: Factorial

def fact(n):
    if n <= 1:
        return 1
    else:
        return n * fact(n - 1)

Calling fact(5) unfolds into a chain: 5 × fact(4) → 4 × fact(3) → 3 × fact(2) → 2 × fact(1) → 1. Each call waits for the next one on the call stack; only when fact(1) returns do the multiplications happen, in reverse.

5.4From Recursion to Recurrences

What is the running time of fact(n)? Each call does constant work c (the comparison and one multiplication) plus the time of the smaller call: T(n) = T(n−1) + c, unwinding as T(n−1) + c → T(n−2) + c + c → … down to T(1). The cost of a recursive function is naturally another recursive expression:

Recurrence for fact(n) T(n) = 1   when n ≤ 1
T(n) = T(n−1) + 1   when n > 1

Equations of this shape are called recurrence relations. Four standard techniques solve them:

  • Substitution method — unroll the recurrence step by step (§5.5).
  • Recursion-tree method — draw the calls as a tree and sum level by level (§5.6).
  • Master method — a ready-made theorem for common shapes (§5.7–5.10).
  • Akra–Bazzi method — a generalisation for unequal splits (beyond this course's scope).

5.5The Substitution Method

Unroll the recurrence until the pattern is clear, then close it with the base case — exactly as we did for binary search in Chapter 3:

T(n) = T(n−1) + c
     = T(n−2) + c + c
     = T(n−3) + c + c + c
     ⋮
     = T(n−k) + k·c

Let n − k = 1  ⇒  k = n − 1
Since T(1) = 1:  T(n) = 1 + (n−1)·c  ⇒  O(n)

Exercises — solve using the substitution method

Q1 — T(n) = T(n−1) + n, with T(1) = 1
show answer
O(n²) 
T(n) = T(n−1) + n
     = T(n−2) + (n−1) + n
     = T(n−3) + (n−2) + (n−1) + n
     ⋮
     = T(1) + 2 + 3 + ⋯ + n
     = n(n+1)/2  ⇒  O(n²)

Each level adds the current n, so the total is the arithmetic series — the same triangular sum behind selection sort's recurrence (Chapter 3).

Q2 — T(n) = T(n−1) + log n, with T(1) = 1
show answer
O(n log n) 
T(n) = T(1) + log 2 + log 3 + ⋯ + log n
     = 1 + log(2 · 3 · ⋯ · n)
     = 1 + log(n!)  ≈  n log n        (Stirling)

The sum of logarithms is the log of the product — log(n!) — which grows as Θ(n log n). A useful identity to remember.

5.6The Recursion-Tree Method

Draw every recursive call as a node labelled with its local cost, sum each level, then sum the levels. Three trees tell most of the story:

A balanced split: T(n) = 2T(n/2) + n

An unbalanced split: T(n) = T(n/3) + T(2n/3) + n

The general picture: T(n) = a·T(n/b) + f(n)

Exercise — write the recurrence for the Fibonacci series and solve it with a recursion tree
def fib(n):
    if n <= 1: return n
    return fib(n - 1) + fib(n - 2)
show answer
O(2ⁿ) 
T(n) = T(n−1) + T(n−2) + c ,   T(0) = T(1) = 1

Tree: every node has 2 children; the shallowest path
(always n−2) has depth n/2, the deepest (always n−1) has depth n.
Number of nodes is therefore between 2^(n/2) and 2^n.

Each node does constant work, so T(n) grows exponentially — between 2^(n/2) and 2ⁿ, i.e. O(2ⁿ). The tree also shows why: fib(n−2), fib(n−3), … are recomputed again and again in different branches — the redundancy that dynamic programming (Unit 4) will later eliminate.

Exercise — T(n) = 3T(n/4) + c·n² (Cormen, pp. 111–112)
show answer
Θ(n²) 
Level i cost:  3^i · c·(n/4^i)²  =  (3/16)^i · c·n²
Total:  c·n² · Σ (3/16)^i  ≤  c·n² · 1/(1 − 3/16)  =  (16/13)·c·n²

The per-level costs shrink geometrically (ratio 3/16 < 1), so the root dominates and T(n) = Θ(n²). When a level-sum forms a decreasing geometric series, the whole tree costs only a constant factor more than its root.

5.7The Master Theorem

For recurrences of the form T(n) = a·T(n/b) + f(n) (a ≥ 1, b > 1), the Master theorem reads the answer off Fig 5.3's tree by comparing f(n) — the root's cost — against n^log⁂a — the leaves' cost:

CaseConditionMeaningSolution
Case 1 f(n) = O(n^(log⁂a − ε)) for some ε > 0 f(n) grows polynomially slower than n^log⁂a (by an nᵗ factor) — the leaves dominate Θ(n^log⁂a)
Case 2 f(n) = Θ(n^log⁂a · lgᶠn) for some k ≥ 0 f(n) and n^log⁂a grow at similar rates — every level contributes Θ(n^log⁂a · lgᶠ⁺¹n)
Case 3 f(n) = Ω(n^(log⁂a + ε)) for some ε > 0, and the regularity condition a·f(n/b) ≤ c·f(n) holds for some c < 1 f(n) grows polynomially faster — the root dominates Θ(f(n))
Intuition Case 1 = the tree is bottom-heavy (leaves win). Case 3 = top-heavy (root wins). Case 2 = perfectly balanced, so the log n levels each pay their share, adding one logarithm to the answer.

5.8Master Theorem: Worked Examples

Ex 1 — T(n) = 4T(n/2) + n
show answer
Θ(n²)

a = 4, b = 2 ⇒ n^log⁂a = n²; f(n) = n. Case 1: f(n) = O(n^(2−ε)) with ε = 1. Therefore T(n) = Θ(n²).

Ex 2 — T(n) = 4T(n/2) + n²
show answer
Θ(n² lg n)

a = 4, b = 2 ⇒ n^log⁂a = n²; f(n) = n². Case 2: f(n) = Θ(n² lg⁰n), that is, k = 0. Therefore T(n) = Θ(n² lg n).

Ex 3 — T(n) = 4T(n/2) + n³
show answer
Θ(n³)

a = 4, b = 2 ⇒ n^log⁂a = n²; f(n) = n³. Case 3: f(n) = Ω(n^(2+ε)) with ε = 1, and the regularity condition holds: 4·(n/2)³ = n³/2 ≤ c·n³ with c = 1/2 < 1. Therefore T(n) = Θ(n³).

Ex 4 — T(n) = 4T(n/2) + n²/lg n
show answer
Master method does not apply

a = 4, b = 2 ⇒ n^log⁂a = n²; f(n) = n²/lg n. It is not Case 1: for every constant ε > 0 we have nᵗ = ω(lg n), so f(n) is not polynomially smaller than n². And it is not Case 2 with k ≥ 0 (it would need k = −1). The classic three-case theorem is silent here — the extended dividing form in §5.10 (with p = −1) handles it: T(n) = Θ(n² lg lg n).

5.9Master Theorem for Decreasing Functions

General form · decreasing function T(n) = a·T(n−b) + f(n),   where a > 0, b > 0, and f(n) = O(nᶠ), k ≥ 0
ConditionSolution
a < 1O(nᶠ)  i.e.  O(f(n))
a = 1O(nᶠ⁺¹)  i.e.  O(n · f(n))
a > 1O(aⁿጫᶦ · nᶠ)  i.e.  O(aⁿጫᶦ · f(n))

Intuition: subtracting b gives a chain of n/b levels. With a = 1 the chain just sums f over n levels; with a > 1 the number of calls multiplies at every level — exponential blow-up.

2) T(n) = T(n−1) + n        5) T(n) = 3T(n−1) + 1
3) T(n) = T(n−1) + log n    6) T(n) = 2T(n−1) + n

a = 1 vs a > 1

With a = 1 (add one level of f each time): 1) O(n), 2) O(n²), 3) O(n log n) — exactly our substitution results from §5.5. With a > 1 and b = 1: 4) O(2ⁿ), 5) O(3ⁿ), 6) O(n·2ⁿ) — the aⁿ term takes over.

Question — complexity of the recursive Fibonacci program via this theorem
T(n) = 2T(n−1) + 1
show answer
O(2ⁿ)

Treating T(n) = T(n−1) + T(n−2) + c as at most 2T(n−1) + 1: a = 2 > 1, b = 1, k = 0 ⇒ O(aⁿጫᶦ·nᶠ) = O(2ⁿ) — agreeing with the recursion-tree answer from §5.6.

5.10Master Theorem for Dividing Functions

General form · dividing function T(n) = a·T(n/b) + f(n),   where a ≥ 1, b > 1, and f(n) = Θ(nᶠ logᵖn), k ≥ 0

Step 1: determine log⁂a and k, then compare:

ComparisonSub-conditionSolution
log⁂a > k—Θ(n^log⁂a)
log⁂a = kp > −1Θ(nᶠ logᵖ⁺¹n)
p = −1Θ(nᶠ log log n)
p < −1Θ(nᶠ)
log⁂a < kp ≥ 0Θ(nᶠ logᵖn)
p < 0O(nᶠ)

O(n log n)

log₂2 = 1 and k = 1 (p = 0): the "equal" row with p > −1 gives Θ(n¹ log⁰⁺¹n) = Θ(n log n) — matching the balanced tree of Fig 5.1. And §5.8's Ex 4 resolves here too: log₂4 = 2 = k, p = −1 ⇒ Θ(n² log log n).

5.12Binary Search: Iterative vs Recursive

The same algorithm, written both ways — recursion replaces the while-loop with self-calls on a half-sized range:

Iterative (Chapter 3)

BinarySearch(array, key):
    1. Set low = 0
    2. Set high = length(array) − 1
    3. While low <= high:
        a. Set mid = low + (high − low) // 2
        b. If array[mid] == key:      return mid
        c. Else if array[mid] < key:  Set low = mid + 1
        d. Else:                      Set high = mid − 1
    4. return −1   // Target not found

Recursive

def binary_search_recursive(arr, low, high, target):
    # Base case: If the range is invalid, the target is not found
    if low > high:
        return -1
    mid = (low + high) // 2
    if arr[mid] == target:
        return mid
    elif target < arr[mid]:
        return binary_search_recursive(arr, low, mid - 1, target)
    else:
        return binary_search_recursive(arr, mid + 1, high, target)
Its recurrence T(n) = T(n/2) + c — one recursive call on half the range. By the dividing form (a = 1, b = 2 ⇒ log₂1 = 0 = k, p = 0): Θ(log n), agreeing with Chapter 3's substitution. Note the trade-off: the recursive version is shorter but spends O(log n) stack memory on its calls; the iterative one uses O(1).

5.14Divide and Conquer

Definition · Divide and conquer A strategy to solve problems by breaking them into smaller subproblems, solving these subproblems recursively, and combining their solutions.

The three steps

  • Divide — split the problem into smaller, independent subproblems.
  • Conquer — solve the subproblems recursively.
  • Combine — merge the solutions of subproblems to get the solution to the original problem.

General algorithmic template

def divide_and_conquer(problem):
    if base_case(problem):
        return base_solution(problem)
    subproblems = divide(problem)
    subresults  = [divide_and_conquer(sub) for sub in subproblems]
    return combine(subresults)

Characteristics a problem must exhibit

  • Subproblem optimality — subproblems are simpler versions of the original.
  • Independent subproblems — subproblems can be solved separately.
  • Merge step — a way to combine the solutions effectively.

Binary search through the D&C lens

  • Divide: split the sorted array into two halves at mid.
  • Conquer: recursively search only the relevant half.
  • Combine: nothing to merge — the result is returned directly.

5.15D&C: Advantages, Disadvantages, and Examples

AdvantagesDisadvantages
Efficiency — reduces the problem size logarithmically and takes advantage of parallelism (independent subproblems can run on different cores). Overhead — recursive calls use extra stack memory.
Simplicity — a clear three-step structure for problem-solving. Combine-step complexity — the merging process can sometimes be inefficient.
Proven track record — sorting (Merge Sort, Quick Sort), searching (Binary Search), matrix multiplication (Strassen’s algorithm). Not always ideal — for problems without optimal substructure, or with overlapping subproblems (those call for dynamic programming instead).
Where next The next chapters put this machinery to work: merge sort and quick sort are divide-and-conquer through and through, and their recurrences fall straight to the Master theorem you now own.
